Getting started

Dependencies

MikeNet GUI is a program written in Python, so it is designed to work on any operating system where Python is installed. In addition to Python, it requires that you have some additional Python modules. The following list includes all required and optional dependencies:

  • Python 2.7 (or higher)*
  • PySide
  • NumPy
  • Matplotlib
  • SciPy
  • Pydot
  • SCons (optional...otherwise, you need to have some version of the 'make' utility installed)
  • Graphviz (optional, for viewing network diagrams)
An easy way to get all dependencies simultaneously is to obtain a Python distribution such as Canopy by Enthought. Make sure the distribution includes the packages listed above.

* MikeNet GUI has not been tested on Python 3. If using version 3, be aware there may be bugs.

Installing MikeNet

In your shell or terminal window, cd into the resources directory of this folder, and run the MikeNet installer program.
This will set up the modified version* of the MikeNet library. To do this, type the following:

python mikenet_installer.py

Follow the instructions, and then you can use the GUI. To open the GUI, use the cd command in your terminal to make sure
you are in the MikenetGUI folder. Then type the following command:

./gui.py

or

python gui.py

This should load the GUI application in a separate Python-owned window. Note, it may be very slow the first time it is
loaded because it has to import several heavy libraries.

* The modification is simply the addition of an SConstruct file to allow SCons as a build option. None of the original MikeNet libraries have been altered.

Setting up preferences

With the GUI open, you can access some important commands from the main menu, located at the top of the screen. Depending on your operating system's look and feel, the menu may be part of the application window itself, or at the top of the display. If using a Mac, the top menu bar may simply say "Python".

You can use the File menu on the menu bar to load and save scripts. For settings that are more permanent, there is a preferences menu. Access this menu by navigating to File->Edit Preferences.



The preferences menu is divided into two sections: Application Settings, and Database Settings.



Application Settings

If you ran the mikenet_installer program, then you should see the path to your MikeNet library in the top line. On the second line, the build method should reflect the tool you used to compile MikeNet. The default is SCons, but Make is allowed as well (this includes gmake). Make sure that the build method is the same that was used to compile MikeNet, or simulations will not compile properly.

If you choose to not use the mikenet_installer program, and you want to use an existing build of MikeNet, then you must set the MikeNet path to the desired location. Again, make sure the build method is appropriate for your build of MikeNet. If you normally use environment variables, or MikeNet is on your system path, you still need to set the path here, as the GUI does not search your path or environment when compiling simulations. In other words, the GUI has its own way of compiling simulations that is totally encapsulated from the rest of your system. For this reason, the GUI's installation of MikeNet will not interfere with any other installations.

The GUI has support, via Python's multiprocessing module, for multiple CPUs. The program does this by "discovering" how many processes are available, and spawning that number of asynchronous workers. These workers then independently compile and execute runs from a shared queue until all runs are complete and all data has been stored. The preferences menu allows you to turn this feature off, which would result in serial execution of runs (if using Windows, multiprocessing has been found to be buggy for this application, so serial mode is recommended).

A maximum number of processors can be specified, which is useful if you do not want multiprocessing to tie up all available CPUs.

The preferences menu also includes a feature that will inform the user via email when scripts have completed. This is optional.

Database Settings

Data generated from simulations are stored automatically in a "raw" form (different files for activation values, test output, weights, errors, and metadata). A cleaner method of organization involves the use of a relational database. The GUI supports the following SQL flavors of database storage: SQLite, MySQL, and PostgreSQL.

In the preferences menu, the database feature can be disabled. If enabled, two pieces of information are required: the database path, and the driver.

The database path must be specified so that the GUI knows what file to store the data in. The path need not exist (especially if setting up for the first time). If the database does not exist, the GUI will automatically create it when it needs to save data. The driver must be among the options listed. The driver must match the type of database, and an error will be raised if there is a mismatch.

The host name, user name, and password, should be used only if they are required to connect to the database. For example, host name for SQLite will always be localhost because SQLite cannot be used remotely. If the database is secured for certain users, then the appropriate login credentials are required.

All preferences are stored in a single file: MikenetGUI/resources/preferences. If the GUI cannot load that file, it will attempt to load a backup file, MikenetGUI/resources/preferences_backup. It is recommended that the preferences files be left alone so they are not corrupted. In the event that the GUI cannot load either the preferences or the backup, then the entire GUI may need to be reinstalled from scratch.

Program Sections

The program is divided into "tabs" which allow the user to work with one facet of the model at a time. The way that tabs are displayed reflects the underlying tree datastructure. The root node of the tree is the script, which has its own tab, called the Script Object tab. The Script Object tab is always accessible from the top of the display, and it is always the farthest to the left. This is essentially the main tab, and it is where the script is executed.

As you move from left to right, tabs are associated with elements deeper in the tree. The Run Object tab, for instance, would be associated with a run, and it would appear one tab to the right (read "one-level deeper"). The leaves of the tree correspond to events. There are no deeper structures, so the Event tab is the rightmost tab. To reduce the number of tabs (especially for large models), the GUI only shows tabs that are relevant to the parts of the model being edited. This means that only one "path" through the tree is displayed at any given time. There may be other tabs that are hidden, and you can always return to the Script Object tab and work your way back to whatever part of the model you want to edit.



Iterator objects also have tabs which allow their properties to be edited. In the model tree, iterators are treated as direct descendents of the script node.

An exception to the above is the "Edit Train/Test Sets" tab, which is accessible at the top-right of the display. Training and Test sets are globally available to the entire script, so they can be seen as an extension of the script node. They are set aside on their own tab to avoid screen clutter.

Script tab

The Script tab (or Script Object tab) is the main tab of the program. The most important elements are the timeline, the "Script name" text box, and the "Start Script" button.


Back to top

Timeline

The timeline is where the user can create and order runs, where "run" is synonymous with a MikeNet simulation. Here, the user can also create and manage iterators for exploration of parameter space.

Iterators can contain nested runs, as well as other iterators. An iterator without a run inside it does nothing. An iterator can only be used with one run at a time, but any number of iterators can be used for a single run.

The "Edit Script Timeline" controls allow for creation, deletion, and duplication of both runs and iterators.

In the timeline itself, objects are shown within tags (<Run>...</Run> or <Iterator>...</Iterator>). The user can interactively relocate an object by clicking anywhere between an object's tags and dragging it. Runs can also be dragged into iterators, which will effectively bind the run to the iterator. To undo, simply click the run and drag it back outside the tags of the iterator.

Script Name

It is required that each execution of the script be named, and this name is used to help organize any generated data.

This script name is not necessarily the same as the filename of the script. For example, you may have a script file called "xor.mnscript". This script can be loaded into the GUI any time that you want to use the xor model. However, each time you actually execute the script, you will want to use a different script name (for instance, you could use the date: "xor 06-16-14"), otherwise any previous data under the old script name would be overwritten.

In short, it is always advised to use a unique script name for each new execution of a script.

Executing a script

The "Start Script" button should only be clicked when all aspects of the model are specified correctly, and you are ready to generate simulation data. The model cannot be trained interactively; in other words you cannot start training, pause, tweak parameters, and then resume. Essentially, you must specify everything using the GUI, and the "Start Script" button compiles your specifications into code which is then executed as a separate application. Of course, you can always refine the script and then rerun it with changes if you need to.

When you click "Start Script", the GUI will first check for blatant errors, such as missing parameters, bad example files, incorrectly built MikeNet, etc. It will not check everything. If it does find an error, it will report the source of the error and suggest fixes where possible. The GUI will not try to execute a script when errors are found.

If no errors were found but the model seems to behave incorrectly, then it is recommended to look at the log file which is generated by the simulation. You can find this log file in MikenetGUI/data under the appropriate folder. The log file will be named after the run you are interested in with an extension of .log. The log file can be useful in knowing how far into training your model actually went. If unexpected runtime errors occur, they should be printed in the log file as well.

Run tab

The Run tab, or Run Object tab, contains all controls relevant to a single MikeNet simulation. A script can contain any number of runs, though in most cases you will only need one.

In the simplest case, a run consists of a single training event (see the XOR tutorial). However, a run can have a much more complex organization, with any number of child events. If using multiple events, then you can set them up to occur either sequentially or in a probabilistic, interleaved fashion. Each event can have its own set of parameters, and/or draw from its own set of examples.

The GUI allows for events to be grouped into "phases". This is useful in the Triangle Model, for instance, where probabilistic interleaving is used to train different subnetworks for some number of trials, then the entire network is trained on a different set of patterns and tested. The first phase of probabilistic interleaving involves four different events (each event has its own set of examples and works on different network components). The second phase involves only a single event and adds testing procedures. Conceptually, all of this occurs as part of a single run.

Some parameters (random seed, bias value, number of time ticks...) are defined only at the run-level, and they apply to all phases and events that are children of that run.

The Run tab has three sub-tabs. They are labelled "Run Properties", "Groups and Connections", and "Events". These are described in detail below.

Back to top

Run properties

This tab allows you to edit run-level properties that will apply to all children of the run (phases and events).

As with the script name, it is recommended that you enter a descriptive run name. However, the run name does not need to be unique from any previous runs you have done. The run name is mainly used to name the output files (a run called "xor" will generate a log file named "xor.log"). You can also use the run name as a variable when extracting data from the database (e.g. if you want all "xor" data).

Random seed, time ticks, weight range (the range for random initialization of connections) and bias value are all run-level parameters. All of these, with the exception of time ticks, can be overwritten by using an iterator. If you do use an iterator for one of these parameters, then whatever values are entered in the run properties tab will be ignored at compile time, and the iterator values will be used instead. See the XOR tutorial for an example of how to iterate over a range of random seed values.

Groups and connections

The Groups and Connections tab enables the user to design the network architecture: the groups (layers) and connections between them.

To create a new group, click "Add Group". To delete, click "Delete Group" (only works if a group is selected). To select a group, click the name of the group along the top row or the far-left column (it does not matter which, because they refer to the same group). When selected, a group name will be highlighted. You cannot change or remove the Bias group.

 To toggle a connection between source group i and destination j, simply click the rectangle in row i, column j of the matrix. An active connection will show up as green, except for connections from the Bias, which show up as orange to distinguish them. All groups are connected to the Bias by default, but these connections can be removed the same way as connections between regular groups.



When a group is selected, then the Group Properties fields become active at the bottom of the tab. Here you can name the group, fix the number of units, and set the activation function type and error computation type.

Clamps and targets are not applied in the GUI. Rather, they must be specified in an external example file according to the MikeNet guidelines. A typical example should look like this:

TAG Word: dog 
PROB 0.2 
CLAMP MyInput 0-2 FULL 
1 0 1 1 0 0 0 , 
TARGET MyTarget 1-2 FULL 
0 0 0 1 1 0 1 , 
;

Given this example, notice that you are required to use the GUI to create one group with the name "MyInput" and one named "MyTarget". The names of the groups must match what is in the example file exactly (names are case sensitive). Furthermore, this example file would also require that MyInput and MyTarget have 7 units each. You need to be familiar with any example files that you are using, and make sure that the name and number of units are set accordingly. The GUI may not be able to catch these kinds of mistakes.

More information on proper example formatting can be found here.

The "Visualize" button may be disabled. If it is, that means Graphviz has not been installed on your system. This is an optional feature. If you install Graphviz, then you may click this button to generate a simple network diagram. This can be helpful in double-checking your work, especially with larger networks.

Events

The "Events" sub-tab is the final editable section when setting up a run. Events are really where all the action happens. An event is a set of definitions, including (but not limited to) training parameters, ONE training example file, and any number of test routines. The event can use only a portion of the network, if desired, leaving the rest of the network out of the weight-updating algorithms. It can also utilize different types of noise.

A run can have any number of events, and these events can be strung together in a sequence (default) or organized into multiple phases.  This allows for complicated behavior including using a combination of probabilistic interleaving and sequential event ordering. A single phase will suffice for most applications.

There are two sections of this tab: Interleaving and multi-phase control, and phase timeline control.

Back to top

Interleaving and multi-phase control panel



By default, a single phase, called "defaultPhase" is provided with every run. More phases can be added as needed by clicking "Add Phase".

Click a phase name or index in the window to select that phase. Selecing a phase will also update the phase timeline control panel (on the right-hand side). It is possible to remove phases, but at least one phase is required at all times.

The phase name can be edited by double-clicking the name.

The Order column can be set to SEQ (sequential order) or PROB (probabilistic order). If using SEQ, then all events in that phase's timeline will occur in-order. If using PROB, then events in that phase's timeline will occur with their respective probabilities (probabilities are set in the individual event's tab).

The Max Iterations column is only relevant when using PROB order. It sets the maximum number of total trials for the phase (where each trial is non-deterministically chosen from the distribution of event sampling probabilities).

Back to top

Phase timeline control panel



The phase timeline is where events are created. If the phase order type is SEQ, then order of events matters. You can reorder events in the phase timeline by clicking an event and dragging it to its new location. If the phase order type is PROB, then the order in the timeline does not matter.

Whether the event mode is TRAIN or TEST is determined by editing the event. The phase timeline only displays the mode (and the p value, if using PROB) to be informative; it does not allow you to directly edit these properties.

To open an event for editing, double-click the desired event. Alternatively, select the event by clicking once, and then click "Edit Selected". This will navigate to the event's tab.

When an event is removed from the timeline, it is removed permanently. Removal of events (or any other data object) is not undoable.

Event properties


The event properties sub-tab enables the main features of the event to be defined.

The event name is an optional descriptive tag for an event.

The proportion value is used only if the event is part of a phase of order type PROB. It represents the probability that this event will be used for any given trial. All event proportions within a phase must add to 1.

The event can be linked to any number of test sets. Clicking "Add Link" will bring up a menu of all available test sets. For more on how to create a test set, see Using test sets.

If the mode is TEST, then this event will not include any training. Instead, all test sets that have been linked to the event will be carried out. If the mode is TRAIN, then testing can still be done intermittently throughout training. The frequency of testing, in this case, is specified by the "test interval" parameter in the event's training set.

Finally, if the mode is TRAIN, then the event will need to be linked to a single training set. Available training sets are listed in the dropdown. Read more about creating training sets here.

Once linked to a training set, all overridable parameters are displayed in a box to the right. This allows you to create only a single training set, and to override parameters specific to an event when needed. Why would this be useful? Consider a run where you want to start with a high learning rate, and then switch to a low learning rate after 1000 trials. All other parameters will be the same. This could be accomplished by having two events. The two events are both linked to the same training set, so all parameters are the same. However, you can override the learning rate in the first event and set it to the desired higher value. You can also override the max iterations value of the first event to be 1000, so the second event will start after 1000 trials.

Isolate network components

 MikeNet is designed to allow sub-networks of large models to be trained independently. This flexibility is built-in to the GUI: a training or test event can be carried out on a subnetwork of your choosing. The "Isolate Network Components" sub-tab uses an adjacency matrix much like in the run tab. However, groups and connections cannot be added or removed from this matrix in the same way. The matrix only displays the available groups and connections for this run, and you can choose which components will be included in this event.

Two examples are shown side-by-side below. Both involve the same network structure (a simple one, with two groups, and one connection). In the image on the left, only "newGroup1" has been included in the event, which we can tell because it is the only one highlighted. In the image on the right, both groups are now included, as well as the connection from "newGroup1" to "newGroup2".
